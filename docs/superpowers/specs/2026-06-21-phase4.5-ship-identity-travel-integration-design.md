# Phase 4.5 — Ship Identity & Travel Integration Design

Date: 2026-06-21
Status: Approved for implementation
Parent spec: `docs/superpowers/specs/2026-06-20-synapse-sea-core-systems-design.md` (System 4 + System 5 seam)
Builds on: Phase 4 scanner/travel logic (`ScannerState`, `Synapse SeaWorld`, `TravelController`, `MarkerGenerator`, `ShipMarker`)
Integrates: Phase 2 ship systems (`ShipSystemsManager`), Phase 3 progression (`scanner_operation` skill), procgen (`ShipGenerator.generate_from_seed`)

---

## Why this phase exists (the gap)

Phases 1–3 are wired into the live game: procgen *is* the playable ship; `ShipSystemsManager`
and `PlayerProgressionState` are owned and driven by the coordinator
`scripts/procgen/playable_generated_ship.gd`. **Phase 4 is not.** `grep` confirms `ScannerState`,
`Synapse SeaWorld`, and `TravelController` appear only in their own files and their own smokes —
nothing in the coordinator or `main.gd` references them. There is no in-game way to scan or travel.

The deeper structural gap: **there is no `ShipInstance` / "current ship" abstraction.** Phase 4's
`TravelController.attempt_travel()` returns a `Node3D` that nobody holds. The coordinator builds
exactly one hardcoded ship with a single `ship_systems_manager` and has no concept of "swap to a
different ship." That keystone is not optional for Phase 5: docking is "attach ship B to ship A," so
a ship must first be a first-class object. Building Phase 5 on un-integrated Phase 4 would stack a
second floating island on the first.

This phase closes that gap with the **minimum that makes travel real**, without rewriting the
2700-line coordinator.

## Goal

Introduce a `ShipInstance` abstraction and wire scanner/travel into the live loop so the player can,
in a running game: open a scanner, see real in-range markers (gated by the ship they are standing
in), select one, and travel — generating that ship and boarding it. Validated headless.

## Scope & boundary

- **In:** `ShipInstance` model; coordinator ownership of `Synapse SeaWorld`/`ScannerState`/
  `TravelController`; `scan()` and `travel_to()` on the coordinator; a real player+camera swap on
  travel; a minimal functional scanner panel with input wiring; headless smokes; one ADR.
- **Out (deferred):** full per-ship state extraction (hazards/objectives/oxygen/route/interactables
  stay on the coordinator — see Approach A vs B below); polished scanner/travel UI and docking
  animation (Phase 7 visual pass); persistent world state for traveled ships (meta-save phase);
  ship-in-ship docking ports/hierarchy (Phase 5 — only declared as stubs here); fuel/time/risk travel
  economy (Phase 6).

### Approach chosen: A — lightweight handle + real swap

Considered three depths:

- **A (chosen):** `ShipInstance` owns only what *must* be per-ship for travel to be real
  (identity, blueprint, its own `ShipSystemsManager`, scene root, Phase-5 stubs). Coordinator gets
  `current_ship` + `travel_to()`. Rich hazard/objective sim stays on the starting ship; traveled
  derelicts are walkable but do not run the full hazard script. Real, headless-provable, no monolith
  rewrite.
- **B (deferred):** extract *all* per-ship state into `ShipInstance`, making the coordinator a thin
  multi-ship manager. Architecturally cleanest and the eventual destination, but a large, risky
  rewrite of the most complex file in the repo — most of which the travel proof does not need.
- **C (rejected):** abstraction seam only, no live swap — does not close the integration gap.

B is the acknowledged long-term target; A is a deliberate, documented stepping stone toward it (the
`ShipInstance` seam is exactly what B would later grow).

## Architecture

```
ShipInstance (RefCounted, pure handle)     scripts/systems/ship_instance.gd
  └─ wraps: ShipSystemsManager (per ship), generated scene_root (Node3D)

PlayableGeneratedShip (coordinator, existing)  scripts/procgen/playable_generated_ship.gd
  ├─ current_ship: ShipInstance              (starting ship wrapped at boot)
  ├─ synapse_sea_world: Synapse SeaWorld           (seeded from the run)
  ├─ scanner_state: ScannerState
  ├─ travel_controller: TravelController
  ├─ ship_generator: ShipGenerator           (injected into travel)
  ├─ scan() -> Dictionary
  └─ travel_to(marker) -> Dictionary

ScannerPanel (Control, on existing HUD CanvasLayer)  scripts/ui/scanner_panel.gd (+ .tscn)
```

### `ShipInstance`

```gdscript
extends RefCounted
class_name ShipInstance

var ship_id: String = ""            # stable per-run id; starting ship uses a fixed id
var marker_id: String = ""          # "" for the starting ship; set for traveled ships
var blueprint                       # ShipBlueprint (size/condition/seed)
var systems_manager                 # ShipSystemsManager — this ship's own systems
var scene_root: Node3D = null       # generated/loaded tree; null when not instantiated

# Phase 5 stubs — declared now so docking attaches to a stable shape; unused this phase.
var parent_ship = null              # ShipInstance | null
var docked_ships: Array = []        # Array[ShipInstance]
var docking_ports: Array = []       # Array (DockingPort in Phase 5)

func get_summary() -> Dictionary    # {ship_id, marker_id, blueprint:to_dict, systems:get_summary}
func apply_summary(summary) -> bool  # round-trips id/marker/blueprint; rebuilds systems via apply_summary
```

- Pure data + a systems handle. No scene-tree logic lives here — the coordinator owns `add_child`/
  `queue_free` of `scene_root`. `ShipInstance` never frees its own root (avoids ownership ambiguity).
- Constructed via a `load(...).new()` self-reference factory if a static builder is added, matching
  the project idiom (`class_name` globals unreliable under `--headless --script`).

### Coordinator integration (`playable_generated_ship.gd`)

**At boot (no behavior change to the starting slice):** after the existing ship + systems manager are
built, wrap them: `current_ship = ShipInstance.new()` with `ship_id = "ship_start"`, `marker_id = ""`,
`blueprint` = the starting blueprint, `systems_manager` = the existing `ship_systems_manager`,
`scene_root` = the existing loader root. The coordinator continues to reference `ship_systems_manager`
directly for all existing hazard/objective wiring (Approach A: starting-ship sim is untouched).
Initialise `synapse_sea_world` (seed from the run/blueprint seed), `scanner_state`, `travel_controller`,
and a `ship_generator` instance.

**`scan() -> Dictionary`:**
```
ops = {
  "navigation": current_ship.systems_manager.is_operational("navigation"),
  "scanners":   current_ship.systems_manager.is_operational("scanners"),
  "propulsion": current_ship.systems_manager.is_operational("propulsion"),
}
skill = player_progression.get_skill_level("scanner_operation")
return scanner_state.scan(synapse_sea_world, ops, skill)
```

**`travel_to(marker) -> Dictionary`:**
```
ops_t = { "propulsion": current_ship.systems_manager.is_operational("propulsion") }
result = travel_controller.attempt_travel(marker, ops_t, synapse_sea_world, ship_generator, scanner_state.range_radius)
if not result.success: return result            # rejection reason surfaced to caller/panel
# success: result.ship is a fresh Node3D root
new_ship = ShipInstance for the marker (own ShipSystemsManager seeded by blueprint condition)
if current_ship.marker_id != "":                # leaving a *traveled* ship (not the starting ship)
    free current_ship.scene_root (queue_free)    # stateless: regenerated from seed if revisited
else:
    remove_child the starting ship root (keep the reference) but DO NOT free it — detaching from the
    tree halts its hazard/oxygen/objective _process so the unoccupied starting ship does not keep
    simulating in the background; its state is frozen intact for a later return
add result.ship under the coordinator; current_ship = new_ship; new_ship.scene_root = result.ship
re-home player + camera to the new ship's spawn (nav-floor spawn, existing spawn logic reused)
return result
```

Invariants:
- The **player node, camera rig, HUD, `player_progression`, and inventory are never freed** across a
  swap — they belong to the player, not the ship. The smoke asserts `player_progression` is the same
  object instance before and after travel.
- `Synapse SeaWorld.set_player_position` + `mark_generated` are performed inside `TravelController` on
  success (already implemented); the coordinator does not duplicate them.
- Returning to a previously-visited marker regenerates the ship deterministically (same seed → same
  layout); any loot/repairs made there are lost. **Documented limitation**, owned by the future
  meta-save/world-persistence phase.

### Minimal scanner panel (`scripts/ui/scanner_panel.gd` + `scenes/ui/scanner_panel.tscn`)

- A `Control` added to the existing HUD `CanvasLayer`, hidden by default.
- Input action **`toggle_scanner`** (added to `project.godot`) toggles visibility; on show it calls the
  coordinator's `scan()` and renders one line per marker: `id · dist · size · <detail-gated fields>`
  using the same view dicts `ScannerState` returns (so detail level is honoured — nav offline shows
  the empty/"no signal" state; higher `scanner_operation` reveals more fields).
- `ui_up`/`ui_down` move a selection highlight; a confirm action (reuse `ui_accept`) calls
  `travel_to(selected_marker)` and shows the result: on success the panel closes; on rejection it shows
  the reason string (`out_of_range`, `propulsion_offline`, `generation_failed`) as a status line.
- Unstyled, functional only. No animation. Polished scanner UI remains a Phase 7 deliverable.

## System integrations (decoupled, matching Phase 4)

The Phase 4 models stay decoupled — they receive plain dicts/values, never import the manager or
progression. The coordinator is the single assembly point that derives `ops`/`scanner_skill` from the
real systems and injects the `ShipGenerator`. This phase only adds that assembly point; it does not
change the Phase 4 model contracts.

## Testing

1. **`ship_instance_smoke.gd`** (RefCounted) — construct a `ShipInstance` with a blueprint + systems
   manager; `get_summary`/`apply_summary` round-trip (`ship_id`, `marker_id`, blueprint fields,
   systems summary); Phase-5 stub fields exist and default empty/null.
2. **`travel_integration_smoke.gd`** (SceneTree — generates real nodes) — build the coordinator on the
   starting ship; `scan()` with navigation online returns markers, with navigation forced offline
   returns `detail_level 0` / empty; pick an in-range marker; `travel_to` with propulsion **offline**
   → `{success:false, reason:"propulsion_offline"}` and `current_ship` unchanged; bring propulsion
   online → `travel_to` succeeds, `current_ship.marker_id` now matches the marker, the new
   `scene_root` is a non-null `Node3D` parented under the coordinator, the player is positioned inside
   it, `synapse_sea_world.is_generated(marker_id)` is true, and `player_progression` is the **same object
   instance** as before travel. Free generated nodes to avoid leak warnings.
3. **`scanner_panel_smoke.gd`** (SceneTree) — instance the panel, point it at a coordinator (or a stub
   exposing `scan()`/`travel_to()`); toggling populates rows from `scan()`; `ui_down` moves selection;
   confirm invokes `travel_to` with the selected marker and renders the result/rejection line.
4. Register all three in `docs/game/06_validation_plan.md` (`commands=58` → `61`); the full bundle must
   end `SYNAPSE_SEA REGRESSION PASS commands=61 clean_output=true`; the Gate-1 automated playtest must
   still pass (the starting-ship slice is unchanged, so it should be unaffected — verified, not

## Risk

Moderate. The new model (`ShipInstance`) is isolated and low-risk. The real risk is the coordinator
swap: re-homing the player/camera, freeing the correct nodes, and never double-freeing or orphaning
the persistent player objects. Mitigations: the SceneTree smoke asserts object identity
(`player_progression` unchanged) and scene parentage after a swap; the starting ship is detached-not-
freed so its persistent sim cannot be corrupted; `ShipInstance` never frees its own `scene_root`
(single ownership at the coordinator). The starting-ship slice and its Gate-1 playtest are
behaviourally untouched, bounding regression surface to the new travel path.
