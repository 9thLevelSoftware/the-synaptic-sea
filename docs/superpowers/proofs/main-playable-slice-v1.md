# Main Playable Slice v1 Proof

## What This Proves

The Main Playable Slice v1 milestone proves that the actual `project.godot` main scene — the coherent proof ship `playable_coherent_ship.tscn` opened via the project's real main path — is now readable and completable end-to-end. From the moment the main scene boots, the player can be spawned, the HUD can read its first objective prompt, four sequential objectives can be completed through actual input, vertical and blocked-route affordances are honored, and the slice completes with the same completion contract used by prior regressions. This proves the slice has graduated from "smoke boots a scene" to "the project's main path is a playable, completable slice."

## Final Acceptance

- [x] **Main path boots into the coherent proof ship.** `main_coherent_boot_smoke.gd` exits with `MAIN COHERENT BOOT PASS scene=playable_coherent_ship objectives=4 critical_path=5 landmarks=2 frames=1`, confirming the main scene loads without errors and exposes four objectives, the critical-path room chain, and both landmarks.
- [x] **Main path's coherent ship is traversable with collision and objective contracts intact.** `coherent_playable_traversal_smoke.gd` exits with `COHERENT PLAYABLE TRAVERSAL PASS rooms_traversed=5 side_rooms=3 blocked_route_blocked=true objective_completed=true`, confirming side-room reachability, blocked-route affordance, and the first objective's interaction contract.
- [x] **Main playable slice HUD renders the first prompt and tracks the active sequence.** `MAIN PLAYABLE SLICE HUD PASS canvas_layer=true width=520 current_sequence=1` confirms the HUD canvas layer attaches at 520px width and reports the active objective sequence.
- [x] **Main playable slice completes all four objectives through real input.** `MAIN PLAYABLE SLICE COMPLETE PASS completed=4 current_sequence=5 run_complete=true` confirms all four objectives (cargo supply cache, maintenance breaker panel, medbay terminal, reactor control panel) resolve through the real interaction path and `run_complete` flips at sequence 5.
- [x] **Main playable slice affordances are present and discoverable.** `MAIN PLAYABLE SLICE AFFORDANCE PASS objectives=4 blocked=1 vertical=1 landmarks=2` confirms four objective affordances, one blocked route, one vertical transition, and two landmarks.
- [x] **Main playable input loop moves the player, follows the camera, and routes input through the interaction path.** `MAIN PLAYABLE INPUT LOOP PASS moved=true camera_followed=true interaction_input_path=true current_sequence=2` confirms movement, camera follow, and the real interaction input pipeline advance the sequence from 1 → 2.
- [x] **Procgen seed-17 playable ship regression still passes alongside the slice.** `PLAYABLE SHIP SMOKE PASS player_spawned=true collision_checked=true interaction_completed=true objectives_completed=1 objective_count=4 collision_shapes=317 frames=1` confirms the procgen playable ship path (which seeds deterministic content and exercises collision at scale, 317 shapes) still completes its first objective without regressing.
- [x] **Capture sequence produces six viewport frames covering the full slice.** `MAIN PLAYABLE SLICE CAPTURE SEQUENCE PASS frames=6 mode=viewport output_dir=.../main_playable_slice_v1` confirms six frames are produced in viewport mode and cover spawn, first prompt, first completion, blocked route, vertical transition, and slice complete.
- [x] **Capture frame for the slice-complete state is written.** `06_slice_complete.png` is produced at 1280×720 and committed to the artifact directory.

> **Note on runtime values.** The numbers in the markers above (`frames=1`, `frames=6`, `collision_shapes=317`, etc.) are recorded values from this validation run — they reflect the state observed when the smokes actually executed against this checkout, not hardcoded constants. ## Limitation Notes

- This is a prototype slice, not a shippable release. Asset fidelity is placeholder, audio cues are not present, and the HUD uses a 520px fixed-width layout rather than a fully reactive UI.
- The capture sequence runs at 1280×720 viewport size and captures whatever the player+camera can see at scripted moments — it is a regression artifact, not a marketing screenshot. Visual polish, lighting passes, and VFX are out of scope for v1.
- The slice is tested against the **coherent proof ship** (`playable_coherent_ship.tscn`), which is the only ship layout that exposes all four critical-path rooms and both landmarks in a hand-authored form. The procgen playable ship (`procgen_playable_ship`) is regression-tested separately for spawn/collision/interaction but does not yet ship the full four-objective sequence.
- Blocked-route and vertical-transition affordances are honored by the slice but are intentionally minimal — one blocked passage and one vertical ladder. The traversal smoke verifies they exist and that the blocked route cannot be traversed; further variety is a v2 concern.
- All input is exercised through the slice's real interaction pipeline. The headless smokes drive it by calling `Player.request_interact()` directly, which emits the `interact_requested` signal; `PlayableGeneratedShip` connects that to `_on_player_interact_requested`, which walks the registered `Interactable` list and calls `try_interact(player_body)` on the first one that completes. The OS/InputMap `Player._unhandled_input` path (`Input.action_press("interact")` → `_unhandled_input` → `request_interact`) is defined and reachable for real human input, but the smokes don't drive it — they invoke `request_interact()` directly to stay headless. No synthetic shortcuts around the interaction contract were added; future slice expansions must extend the same `request_interact` → `interact_requested` → `_on_player_interact_requested` → `Interactable.try_interact` chain rather than introduce a parallel one.
